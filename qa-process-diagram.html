QA Process Improvement Proposal

Transitioning to a Preventive, Mini Test Plan Approach

1. Current Context and Challenges

At present, our QA process operates reactively within our team. Acceptance criteria for tickets are often defined after development completion, based on what developers have already implemented.

This creates specific challenges for small teams:

Communication Gaps

  • QA validation is limited to confirming coded functionality rather than ensuring alignment with Product Manager's original vision
  • No standardized documentation prevents our team from having shared reference points
  • Edge cases are not identified from specs

2. Objective: Mini Test Plan Approach

The objective is to implement a lightweight, specification-first QA approach optimized for the team.

Why "Mini" Test Plans?

Tailored for Small Teams:

  • Comprehensive enough to prevent major issues
  • Concise enough for one QA to manage multiple features simultaneously
  • Quick enough to fit our development pace

Strategic Focus:

  • Cover critical happy paths and high-risk edge cases
  • Prioritize test scenarios that matter most for product success
  • Improve collaboration between Product, Design, Engineering, and QA

Key Goals:

  • Early Integration: QA begins immediately when specifications are completed
  • Risk Prevention: Identify and mitigate issues before project starts
  • Team Alignment: Create shared understanding across all team members
  • Sustainable Process: Maintain development velocity while improving quality

This proposal does not place QA above Product or Engineering, nor does it mean developers must work at the pace or command of QA

3. Mini Test Plan Workflow

3.1

Specification Review (PRD)

Who: QA

When: As soon as specs are drafted

Output: Initial understanding and clarification questions

3.2

Mini Test Plan Creation

Focus Areas:

  • 3-5 critical happy path scenarios
  • 2-3 high-impact edge cases
  • 1-2 error handling scenarios

Format: Streamlined template

Traceability: Direct mapping to acceptance criteria

3.3

Team Validation

Participants: All team members

Agenda:

  • QA presents Mini Test Plan
  • Engineering validates feasibility (Optional)
  • Product confirm business alignment

Outcome: Aligned understanding before the project begins

3.4

Parallel Execution

  • Test cases execute as features become available in staging
  • Continuous feedback loop with developers
  • Issues identified and resolved within the same sprint

4. Implementation (3 Phases)

Phase 1: Preparation

  • QA reviews the Specs. (PRD)
  • QA creates the Mini Test Plan, focusing on happy path scenarios, edge cases, and error handling scenarios
  • The plan is shared with the team in a lightweight format (e.g. through Slack)

Phase 2: Parallel Execution

  • When the devs have their tickets ready, testing will be performed on the Preview URL as it has always been done
  • As features become available in staging, QA executes the defined test cases
  • Feedback is provided to developers immediately, ideally the same day
  • The Mini Test Plan is refined if new gaps or scenarios are identified during development

Phase 3: Final Validation

  • QA performs a minimal regression covering the most critical flows
  • QA confirms that reported issues have been fixed
  • QA delivers a brief exit summary (2–3 key points: what was tested, main outcomes, and any remaining risks)

Outcome:

This approach integrates QA from the beginning without creating heavy processes. The workflow ensures early prevention, continuous alignment, and confident releases while maintaining team velocity.

7. QA Ticket Flow

QA tickets should follow the flow below:

Acceptance State

  • Tickets that reach the Acceptance state are ready to be tested
  • Engineers who leave their tickets in this state should create the QA ticket and assign it to Sebastián

QA Failed State

  • When a ticket is rejected, it will be moved into the QA Failed column
  • Engineers should move back into In Progress when they work fixing any issue reported by QA
  • When all issues are fixed, the ticket is moved back to the Acceptance column, and a new sub-task is created for QA

QA Success State

  • When a ticket is approved it will be moved into QA Success column
  • The QA team in Linear will be used to create QA task tickets, such as writing an automated test

8. Additional

Bug tickets

The tickets generated when a bug is found should maintain their functionality without changes.

Acceptance Criteria

The Acceptance Criteria of the tickets are written by the devs. This workflow remains the same.

QA should be involved in the review of the ACs:

Reviewing the ACs to ensure:

  • They are measurable and clear

Proposing improvements:

  • QA can identify gaps (e.g., "What happens if the user enters invalid data?", "What happens if the connection is lost?")

5. Benefits

For Developers:

  • Early Feedback: Issues found while context is fresh

For QA:

  • Strategic Impact: Shift from reactive to preventive approach
  • Efficient Resource Use: Focus on high-impact scenarios

6. Why Mini Test Plans Work for Small Teams

Resource Efficiency:

  • Parallel execution prevents end-of-sprint bottlenecks
  • Focused testing maximizes impact of limited QA resources

Communication Enhancement:

  • Single source of truth for quality requirements
  • Visual clarity for technical team members

Important Note:

This proposal does not place QA above Product or Engineering, nor does it mean developers must work at the pace or command of QA. Instead, QA acts as a partner from the start, ensuring alignment and risk prevention. Decisions about QA documentation and practices will be made jointly, strengthened by the feedback provided by the team.

This way, project risks are managed as a whole, rather than in isolated parts.
